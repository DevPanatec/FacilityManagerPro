<h2 id="development-tips" class="group scroll-mt-24">Development tips<a href="#development-tips" aria-hidden="true" class="ml-2 opacity-0 group-hover:opacity-100 transition"><span aria-hidden="true">#</span></a></h2>
<p>Here are a few recommendations when developing Edge Functions.</p>
<h3 id="skipping-authorization-checks" class="group scroll-mt-24">Skipping authorization checks<a href="#skipping-authorization-checks" aria-hidden="true" class="ml-2 opacity-0 group-hover:opacity-100 transition"><span aria-hidden="true">#</span></a></h3>
<p>By default, Edge Functions require a valid JWT in the authorization header. If you want to use Edge Functions without Authorization checks (commonly used for Stripe webhooks), you can pass the <code>--no-verify-jwt</code> flag when serving your Edge Functions locally.</p>
<div class="ch-codeblock not-prose " data-ch-theme="supabase"><div class="ch-code-wrapper ch-code" data-ch-measured="false"><code class="ch-code-scroll-parent"><br/><div><span class="ch-code-line-number">_<!-- -->10</span><div style="display:inline-block;margin-left:16px"><span>supabase functions serve hello-world --no-verify-jwt</span></div></div><br/></code></div></div>
<p>Be careful when using this flag, as it will allow anyone to invoke your Edge Function without a valid JWT. The Supabase client libraries automatically handle authorization.</p>
<h3 id="using-http-methods" class="group scroll-mt-24">Using HTTP methods<a href="#using-http-methods" aria-hidden="true" class="ml-2 opacity-0 group-hover:opacity-100 transition"><span aria-hidden="true">#</span></a></h3>
<p>Edge Functions support <code>GET</code>, <code>POST</code>, <code>PUT</code>, <code>PATCH</code>, <code>DELETE</code>, and <code>OPTIONS</code>. A Function can be designed to perform different actions based on a request&#x27;s HTTP method. See the <a href="https://github.com/supabase/supabase/tree/master/examples/edge-functions/supabase/functions/restful-tasks">example on building a RESTful service</a> to learn how to handle different HTTP methods in your Function.</p>

<h3 id="naming-edge-functions" class="group scroll-mt-24">Naming Edge Functions<a href="#naming-edge-functions" aria-hidden="true" class="ml-2 opacity-0 group-hover:opacity-100 transition"><span aria-hidden="true">#</span></a></h3>
<p>We recommend using hyphens to name functions because hyphens are the most URL-friendly of all the naming conventions (snake_case, camelCase, PascalCase).</p>
<h3 id="organizing-your-edge-functions" class="group scroll-mt-24">Organizing your Edge Functions<a href="#organizing-your-edge-functions" aria-hidden="true" class="ml-2 opacity-0 group-hover:opacity-100 transition"><span aria-hidden="true">#</span></a></h3>
<p>We recommend developing “fat functions”. This means that you should develop few large functions, rather than many small functions. One common pattern when developing Functions is that you need to share code between two or more Functions. To do this, you can store any shared code in a folder prefixed with an underscore (<code>_</code>). We also recommend a separate folder for <a href="/docs/guides/functions/unit-test">Unit Tests</a> including the name of the function followed by a <code>-test</code> suffix.
We recommend this folder structure:</p>
